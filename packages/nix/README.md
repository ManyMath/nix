# nix

A CLI for reproducible Dart and Flutter builds with [Nix](https://nixos.org/).

## Installation

```sh
dart pub global activate nix
```

## Quick Start

```sh
# In your Flutter project root:
nix_dart init linux android web
nix_dart setup
nix_dart shell linux
nix_dart build web
```

## Workflows

### CLI Only

Use `nix_dart` as the source of truth and let it manage `nix.yaml`,
`flutter_version.env`, and `android_sdk_version.env`:

```sh
nix_dart init linux web
nix_dart setup
nix_dart shell linux
nix_dart build web
```

### Vendored Subtree + CLI

If the project already vendors the toolkit, import that layout instead of
creating a second copy:

```sh
nix_dart init --from-existing
nix_dart sync
```

### CLI To Standalone Toolkit

`eject` materializes the shell-script toolkit so the project can keep running
without a Dart global package install:

```sh
nix_dart eject --output-dir tooling/nix
bash tooling/nix/bootstrap.sh
```

## Migration From A Vendored Toolkit

If your project already has `flutter_version.env`, `android_sdk_version.env`,
or a git subtree copy of this repo, import that layout instead of starting from
scratch:

```sh
nix_dart init --from-existing
nix_dart sync
```

`nix_dart` uses `nix.flake` from `nix.yaml` as the source of truth. In a host
project with a vendored subtree at `nix/`, that typically becomes `nix/nix`.
The CLI derives the vendored toolkit root from that path, so it can find
`nix/scripts/*`, `nix/Makefile.inc`, and `nix/bootstrap.sh` when present.

## Commands

| Command | Description |
|---------|-------------|
| `init [platforms...]` | Create `nix.yaml` and a starter `nix/flake.nix` |
| `init --from-existing` | Import existing `.env` files and detect a vendored toolkit |
| `setup [platforms...]` | Fetch the Flutter SDK and platform-specific SDK components |
| `shell <platform>` | Enter the configured Nix dev shell |
| `build <platform>` | Build inside the configured Nix shell |
| `doctor` | Check Nix, flakes, `nix.yaml`, `flake.lock`, and vendored toolkit paths |
| `pin` | Update `flake.lock` to the latest nixpkgs revision |
| `sync` | Write `flutter_version.env` and `android_sdk_version.env` from `nix.yaml` |
| `eject` | Generate a standalone toolkit (`bootstrap.sh`, `Makefile.inc`, `nix/`, `scripts/`) |
| `clean` | Remove fetched SDKs and build artifacts |

## `nix.yaml` Schema

```yaml
project:
  name: my_app

flutter:
  version: "3.38.2"
  channel: stable
  checksums:
    macos_x64: ""
    macos_arm64: ""
    linux_x64: ""
    linux_arm64: ""

nix:
  flake: third_party/nix-flutter/nix

platforms:
  linux:
    shell: linux
    build:
      command: flutter build linux --release
      output: build/linux/x64/release/bundle/

  android:
    shell: android
    sdk:
      cmdline_tools_build: "14742923"
      cmdline_tools_sha256_linux: ""
      cmdline_tools_sha256_macos: ""
      platform_version: android-34
      build_tools_version: "35.0.0"
      ndk_version: "28.2.13676358"
    build:
      command: flutter build apk --release
      output: build/app/outputs/flutter-apk/app-release.apk

  web:
    shell: web
    build:
      command: flutter build web --release
      output: build/web/
```

## Notes

- CLI-only projects use the starter `flake_template.dart` generated by `init`.
- Vendored projects should prefer `init --from-existing` so the CLI points at
  the existing subtree flake and scripts instead of creating a second toolkit.
- `sync` is the bridge between the config-driven CLI and the repo-shipped shell
  scripts used by `Makefile.inc`.
- `eject --output-dir path/to/out` writes a self-contained toolkit, using the
  vendored files when present and falling back to the package-bundled toolkit
  assets for CLI-only projects.
- The generated `bootstrap.sh` supports toolkits that live at the project root
  or inside nested vendor directories by searching upward for the host
  `pubspec.yaml`.
